# Документация по работе над проектом

## 1. Воркфлоу

### 1.1 Создание задач

- При создании задачи необходимо указать понятное название, отражающее суть задачи.
- Необходимо указать описание задачи, если название не до конца раскрывает суть задачи или необходимо добавить конкретику.
- Для задачи необходимо указать приоритет (_Priority_):
  - P0 - наиболее приоритетная задача, считай блокер.
  - P1 - задача, которую стоит выполнить в ближайшее время.
  - P2 - наименее приоритетная задача.
- Для задачи необходимо указать ожидаемую сложность (_Size_)
  - Если сложность задачи L, то стоит попробовать разбить её на более маленькие задачи.
  - Если сложность задачи XL, то **необходимо** разбить её на задачи поменьше.
- В столбце Ready находятся те задачи, которые соответствуют правилам и их уже можно взять в работу.

---

### 1.2 Задачи с подзадачами (sub-issues)

- У задачи может быть только один родитель, поэтому стоит аккуратнее отслеживать зависимости между задачами.
- Если задача является необходимой для выполнения другой задачи, её стоит пометить как дочернюю (_sub-issue_).
- Задачи с подзадачами можно брать только в том случае, если все подзадачи были выполнены, то есть прогресс бар родительской задачи показывает 100%

---

### 1.3 Взятие задачи в работу

- Если задача находится в работе, её необходимо перенести в столбец In progress.
- Работа над задачами происходит в отдельных ветках.
- Перед названием ветки задачи необходимо указать тип задачи:
  - `feature/` - работа над фичей приложения.
  - `build/` - задачи, связанные с билдом приложения.
  - `fix/` - задачи, которые исправляют какой-то код.
  - `chore/` - прочие задачи.
- Ветку необходимо создать сразу в удалённом репозитории и сделать чекаут к ней локально.
- Если ветка была создана локально, то с первым же коммитом её необходимо запушить в удалённый репозиторий, чтобы к этому коду в случае чего могли обратиться остальные члены команды.

---

### 1.4 Ревью задачи

- Если задача была выполнена, то её необходимо отправить на ревью, создав Pull Request.
- В канбане необходимо переместить задачу в столбец Review.
- На один Pull Request приходится одна задача.
- В описании Pull Request необходимо указать, что было проделано. Чем детальнее, тем лучше.
- Ревьюить задачу можно даже в случае, если Pull Request не проходит проверку detekt.
- Если Pull Request собрал два аппрува, его можно сливать с веткой.
- Мёрдж веток происходит через **Squash and merge** - все проделанные коммиты сливаются в один.
- После слияния задачу необходимо перенести в Done и удалить рабочую ветку.

Почему так:
- Меньше кода в ревью.
- Меньше merge-конфликтов.
- Чёткая история изменений.

---

## 2. Стиль кода

### 2.1 Самодокументируемость кода

- Код должен быть максимально самодокументируемым.
- Названия классов, функций и переменных должны отражать их назначение.
- Комментарии используются только в случае реальной необходимости.

Допустимо:
- `KDoc` для публичных сущностей.
- `TODO()` (стандартная функция Kotlin) вместо обычных комментариев-заглушек.
- Обычные комментарии — только если без них код становится трудно читаемым.

Недопустимо:
- Комментарии, дублирующие очевидное поведение кода.
- Закомментированный код (его следует удалять).

---

### 2.2 Preview

- Для каждого `@Composable` с превью необходимо указывать аннотации:
    - `@Preview`
    - `@PreviewLightDark`
- При использовании превью необходимо применять `@PreviewParameter` и `PreviewParameterProvider`.
- В `PreviewParameterProvider` должны быть отражены все возможные состояния.

Это позволяет:
- Проверять UI во всех сценариях.
- Упростить тестирование состояния интерфейса.
- Избежать проблем с отрисовкой.

---

### 2.3 ViewModel и MVI

- Все `ViewModel` должны наследоваться от `MviViewModel`.
- Все состояния должны наследоваться от `MviState`.
- Все события должны наследоваться от `MviEvent`.
- Все одноразовые эффекты должны наследоваться от `MviEffect`.

Дополнительно:
- `ViewModel` не может быть параметром `Composable`.
- В `Composable` передаются только:
    - состояние,
    - события (callbacks),
    - необходимые данные.

Это позволяет:
- Переиспользовать экран без привязки к конкретной ViewModel.
- Избежать проблем с Preview.
- Сделать UI слой независимым.

---

### 2.4 Работа с ресурсами

- Все строки, размеры и цвета должны быть вынесены из кода.
- Обращение к цветам происходит через токены `MaterialTheme`.
- Обращение к текстовым стилям происходит через `MaterialTheme.typography`.
- Если строка не отмечена как непереводимая в `strings.xml`, она обязана иметь перевод на английский язык (помимо русского).
- В строках следует минимизировать использование `\n`.
    - Вместо этого необходимо использовать инструменты UI:
        - ограничение ширины,
        - отступы,
        - переносы,
        - layout-компоненты.

---

### 2.5 Иконки и иллюстрации

- Загружаемые иконки должны быть размером **24dp**.
- Название иконки должно быть абстрактным и отражать:
    - изображение на иконке  
      или
    - абстрактное действие.

Пример:
- ✅ `ic_add_24` (на иконке изображён плюс)
- ❌ `ic_fab_24`

Формат именования:
- Иконки: `ic_name_size`
- Иллюстрации: `img_name`

Примеры:
- `ic_add_24`
- `ic_arrow_back_24`
- `img_empty_state`
